A microfrontend is an architectural style where the frontend is split into multiple smaller, independent apps, each responsible for a specific business domain or feature. Each microfrontend: Is built and deployed independently Can use different frameworks (e.g., one in React, another in Angular) Communicates with others at runtime For the same e-commerce site: /home → built with React /cart → built with Angular /checkout → built with Vue /profile → built with React + TypeScript Benefits: Independent deployments (each team can release without breaking others) Faster development cycles (parallel teams) Technology freedom (React, Angular, Vue can coexist) Easier to scale large teams Problems Complex setup and routing between microfrontends Higher infrastructure and build complexity Tools Commonly Used in Microfrontends Nx Monorepo → Organizes multiple apps/libs in one repo Single-SPA → Orchestrates multiple frameworks Webpack Module Federation → Shares modules dynamically WORKING Inside create moduleFederation.config.js file(created in root folder of common repo) https://www.youtube.com/watch?v=FGAcmMufzVU const {dependencies} = require("./package.json"); mode.exports={ name :"", // filename :"remoteEntry.js", //name of the file that will be created by weback(contains info about exposed modules and shared dependencies) remotes : {} // used to specify which remote microfrontends it consumes exposes : {} // which modules mf exposes to others } //ONLY CONTAINER consumes home and course page remotes:{ mainPage:"mainpage@http://localhost:3000/remoteentry.js" coursesPage: } A monolithic frontend means your entire application’s UI is built, deployed, and maintained as one single unit — one codebase, one build, one deployment pipeline.
